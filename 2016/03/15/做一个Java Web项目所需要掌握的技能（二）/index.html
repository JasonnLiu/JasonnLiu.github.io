做一个Java Web项目所需要掌握的技能（二）

发表于 2016-03-15

上一篇，给大家讲述了最基本的java web项目构成，掌握了上一篇的知识，只能算入门，这一篇介绍更多Java web高级技术。

我们先来简单回顾一下上一篇的内容，一个Java web 项目必须要有应用代码，代码得部署到应用服务器上，为了保存项目的数据，还需要一个数据库，比如MySQL，有的项目可能还需要一个文件系统来保存数据。这是一个最简单的web项目架构。


上图表示，应用服务器，数据库，文件系统全都部署在一台主机上。

上述架构最简单的架构，常见于个人项目里，纯属个人兴趣来搭的系统。如果我们的网站或应用非常火爆，有很多人来浏览使用。网站应用的流量增大，越来越多的数据也导致存储空间不够，上述架构是很难承受大的访问压力和存储压力。所以我们必须采取一些措施来改善项目架构。

最直接高效的措施就是将应用和数据分离，分别部署应用服务器，数据库服务器，文件服务器。增加更多的主机，用硬件资源来支撑更大的访问量，每台主机都有自己的角色任务，可以根据任务来定制资源，比如应用服务器需要更强大的CPU，而数据库服务器需要更快更大的硬盘和内存。

当流量进一步增加，我们不断地增加主机的话，不经会增加部署难度，比如主机之间的通信，而且还会增加成本。根据二八定律，80%的业务访问集中在20%的数据上。由此，我们可以利用缓存技术，将一小部分集中访问的数据缓存到内存中，不仅可以分担数据库的访问压力，由于内存访问要明显快于硬盘访问，还能提升访问速度。常用的缓存技术有memcache,redis等。

另外，基于缓存的思想，我们还可以利用反向代理技术和CDN加速来改善应用架构。这两种技术的基本原理都是缓存，但是区别在于CDN部署在网络提供商的机房，用户从距离最近的网络提供商机房获取数据；而方向代理则部署在应用的中心机房，用户首先访问的是反向代理服务器，如果其中缓存了用户所需要的数据，就直接返回给用户，而不会在访问后端的数据库。反向代理常用的软件有Nginx 和 Squid。还记得我们在上一篇也提到过Nginx,它是一个web服务器，我们也可以用其来实现反向代理。

如果流量继续增大，上述技术手段仍不能满足高流量，高并发，需要进一步改善架构。常用的手段还有使用集群。集群是指使用多台主机组成一个主机群来对外提供服务。通过增加或减少集群中主机数量可以动态调整系统性能，从而实现系统的可伸缩性。对于集群来说，非常重要的一项技术就是负载均衡。负载均衡是指访问请求能较为均衡的被指派到集群中的各个主机上。有了集群，就必要要有集群管理技术，Zookeeper就是这样一个框架，帮助我们来管理集群。

一旦使用集群，就进入了如今服务端技术比较热门且重要的领域，分布式。分布式领域是个非常庞大的领域，涉及到许多技术。像分布式存储系统，分布式计算框架等等。

随着应用系统的复杂度的增长，应用代码全都杂糅在一团，耦合性过强。更新代码后，重新部署的时间也原来长，越困难。这时需要将业务进行拆分，功能相近的业务模块，可以抽象出共同依赖的子服务。从而将系统划分为一个个更为精细的子服务，服务内部较为独立，服务之间相互依赖，可以独立部署。服务之间的调用需要利用分布式服务框架，常用的比如Dubbo。

在分布式系统中常用的还有消息中间件，MQ（Message Queue），是一种应用程序对应用程序的通信方法。在项目中，将一些无需即时返回且耗时的操作提取出来，进行了异步处理，而这种异步处理的方式大大的节省了服务器的请求响应时间，从而提高了系统的吞吐量。常见的有ActiveMQ，RabbitMQ。

建设一个成功的互联网应用所用的技术是纷繁复杂，要一一列出几乎是不可能的。希望初学者能在大概了解一些服务端技术之后，挖掘出自己的兴趣所在，深入研究，在一个方向上寻求突破。
